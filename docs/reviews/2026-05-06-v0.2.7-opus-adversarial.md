# Adversarial review — karpathy-wiki at v0.2.7 (post v2.4 split)

Scope: things prior reviews did NOT enumerate. The known-bug list (Codex's 9 items, the field-test bugs, the v1-vs-current diff) is taken as given and not repeated.

---

## 1. Runtime defects nobody has flagged

### 1.1 Spawn double-claim race: `wiki capture` writes the capture, then spawns. Spawn re-claims.
`bin/wiki:258` finishes writing the capture file at `${capture_path}` (a plain `.md`), then `bin/wiki:260` calls `wiki-spawn-ingester.sh "${target_wiki}" "${capture_path}"`. Inside the spawner, `wiki-spawn-ingester.sh:33-42` sees the path does NOT end in `.processing` and tries to `ln "${capture}" "${capture}.processing"`. If a parallel SessionStart drain (running because the user just opened a second terminal) gets there first, the spawner's `ln` fails and it logs `capture already claimed, skipping` and exits 0. **The capture is not lost** (the other drain has it), but `wiki capture` reports `wiki capture: 1 target(s)` to the user as if it had successfully kicked off ingest, when it actually no-op'd. The user has no way to tell. (`bin/wiki:274`)

### 1.2 `set -e` in the spawner combined with `wiki_capture_claim` fallback dead code.
`wiki-spawn-ingester.sh:10` sets `set -e`. The skill at `karpathy-wiki-ingest/SKILL.md:252` says "If `${WIKI_CAPTURE}` is unset or missing, call `wiki_capture_claim` as fallback." But the spawner ALWAYS sets `WIKI_CAPTURE=${processing}` (line 62), and exits 0 before reaching the ingester if it can't claim. The fallback path is unreachable from the spawn flow; the only way to hit it is a user manually invoking `claude -p` with the ingest prompt. The skill prose claims a safety net that does not exist in practice. Either drop it or document it as "manual invocation only."

### 1.3 `_drain_pending` in the hook does NOT pre-claim; `wiki_capture_claim` is unused.
Compare hook `_drain_pending` at `hooks/session-start:359-377` with `scripts/wiki-capture.sh:5-35`. The hook iterates `*.md` (excluding `*.processing`) and calls the spawner directly without first hard-link-claiming. Two SessionStarts firing in parallel (e.g. user opens two terminals at the same instant in the same wiki) BOTH iterate the same set of `.md` files and BOTH call the spawner. The spawner's atomic `ln` makes the actual claim, so only one ingest fires, but every loser SessionStart still spawns N background processes that all immediately log "already claimed, skipping" and die. With N=20 captures and 2 simultaneous SessionStarts that's 20 dead `claude -p` invocations consuming process slots — small but real. The fix is for `_drain_pending` to use `wiki_capture_claim` first (the helper that exists in `scripts/wiki-capture.sh` for exactly this purpose and is currently unused by anyone).

### 1.4 Manifest lock acquired in the hook is process-local, but the hook is a one-shot script.
`hooks/session-start:241-275` opens fd 8 in the parent process for `flock`. After the recovery, fd 8 is closed via `exec 8>&-` (line 280). However the lockfile path is `${wiki}/.locks/manifest.lock` and the inline `_manifest_lock_acquire` function is duplicated in the hook even though `scripts/wiki-manifest-lock.sh` exists for this exact purpose. If those two implementations ever drift on stale-lock policy (currently both 300s), recovery and ingest will fight over the same lockfile with subtly different rules. The hook should call the script.

### 1.5 `wiki ingest-now` recursively invokes the SessionStart hook.
`scripts/wiki-ingest-now.sh:34` literally `cd`s into the wiki and invokes `bash "${REPO_ROOT}/hooks/session-start"`. This is clever (don't repeat code) but means: the loader-emit branch (lines 52-86 of the hook) writes JSON to stdout, which is then consumed by `wiki-ingest-now.sh:34`'s `>/dev/null 2>&1`. Fine today. But if a future hook step prints anything Claude-Code-relevant on stdout (a status request, a hint), `wiki ingest-now` will silently swallow it. Worse, the hook's loader injection emits hookSpecificOutput JSON regardless of the calling context — `wiki ingest-now` is generating well-formed Claude-Code session-start output that lands in `/dev/null`. Wasted CPU on every invocation.

### 1.6 `_drift_scan` proceeds even when manifest lock cannot be acquired during raw-recovery.
`hooks/session-start:300-303`: if `_manifest_lock_acquire` fails, the function returns 0 (no-op). The raw file stays in `raw/` unmanifested. The next SessionStart will see it again and try recovery again. There is no retry counter and no escalation; an unmanifested file is permanently flagged as drift on every session start until manually intervened. `wiki status` does not surface "raw file stuck in recovery loop."

### 1.7 Spawner leaks `ANTHROPIC_API_KEY` to log on macOS.
`scripts/wiki-spawn-ingester.sh:60` puts `ANTHROPIC_API_KEY=${ANTHROPIC_API_KEY:-}` in `env_passthrough[]`, then `setsid env "${env_passthrough[@]}" ${headless_cmd} "${prompt}" >> "${logfile}" 2>&1`. If `claude` ever echoes its env back on startup error (some wrappers do), the secret hits `.ingest.log` which is gitignored at the wiki root but **not** at the `~/wiki/` level if the user has somehow committed it. Defense in depth: pass the key via stdin or via a temporary env file with `0600` perms, not as a shell-quoted env var.

### 1.8 `headless_cmd` is unquoted.
`scripts/wiki-spawn-ingester.sh:71`: `${headless_cmd} "${prompt}"`. Word-splits the value of `platform.headless_command` from `.wiki-config` (the config says `headless_command = "claude -p"`, so it splits to two tokens — that's intentional). But this means any user putting `headless_command = "/path/with spaces/claude -p"` in their config gets a broken spawn with no clear error. There's no validation; `wiki status` does not check that `platform.headless_command` actually resolves to an executable.

### 1.9 `wiki capture` body trap leaks the body file across `exec`.
`bin/wiki:69-70`: `body_buf="$(mktemp)"; trap 'rm -f "${body_buf}"' EXIT`. Both interactive bootstrap branches (`bin/wiki:134` and `bin/wiki:161`) `exec` back into `bin/wiki capture`. Bash's EXIT trap fires before `exec` replaces the process, but the new `exec`'d process then `mktemp`s a fresh file. The OLD body file gets cleaned up correctly. BUT — `--body-file "${body_buf}"` is the path being re-passed; the cleanup races with the re-exec'd process trying to read it. On macOS with TMPDIR on /var/folders this normally works because `unlink` is atomic and the re-exec'd process opens the file before the trap fires. On Linux with extra `inotify` or AV scanning in the path, this is racy. Trace: write file, register trap, exec, trap fires, re-exec'd process opens file. The order between "trap fires" and "child opens" is unspecified.

### 1.10 `_emit_legacy_drift_capture` and `_emit_raw_direct_capture` use `( set -C; : > "${capture_path}" )` for atomicity, then `cat > "${capture_path}"` to write. Window between the touch and the cat.
`hooks/session-start:177-198` and 211-234. The atomic part is the `: > path` (creates exclusively). But the actual content is then written by the subsequent `cat > path <<EOF` (line 178). Between line 177 and line 178, another process scanning `.wiki-pending/` sees an empty `.md` file. The `_drain_pending` loop a few lines later (line 370) treats empty files as captures and spawns ingesters on them. The ingester then chokes on missing frontmatter.

In practice the window is microseconds and the same hook handles both, but: a parallel `wiki ingest-now` (which re-invokes the hook) firing in the gap will see the empty file. Fix: write to `${capture_path}.tmp` and `mv` to `${capture_path}`.

## 2. Unenforceable contracts

### 2.1 `bin/wiki capture --suggested-action` accepts arbitrary strings.
`bin/wiki:60` parses `--suggested-action "$2"` with no validation. The capture-schema lists `create | update | augment | auto` (`capture-schema.md:22`). `bin/wiki capture --suggested-action wibble` writes a frontmatter that the validator does not check (the validator is for pages, not captures). The ingester is on the honor system.

### 2.2 `wiki capture` always writes `evidence: "conversation"` for every kind, including `chat-attached`.
`bin/wiki:239-241` hardcodes `evidence="conversation"` regardless of `--kind`. But `capture-schema.md:60-64` says: *"For `capture_kind: raw-direct` and `chat-attached` → absolute path to the file on disk. NEVER the string `"conversation"`."* The schema demands an absolute path; the CLI cannot produce one (no `--evidence-path` flag). Two consequences:
  - Codex flagged this for `chat-attached`. It is also wrong for `raw-direct` if anyone ever invokes `bin/wiki capture --kind raw-direct` (the help text doesn't list it as user-facing but the flag is accepted at line 91).
  - The ingester's `origin` Iron Rule (`karpathy-wiki-ingest/SKILL.md:291`) says `origin` MUST be the capture's `evidence`. So the manifest entry for any chat-attached capture's evidence file gets `origin: "conversation"`, not the path. The manifest becomes useless for tracing chat-attached evidence back to its source file.

### 2.3 The "thin-capture rejection" loop has no fail-counter.
`karpathy-wiki-ingest/SKILL.md:239-248` says rename `.md.processing` back to `.md` and exit. If the main agent re-spawns without expanding the body, the next ingester rejects again. There is no MAX_REJECTIONS counter. A buggy agent can ping-pong a capture forever, blocking other captures behind it on every drain. (Drain is parallel, so this isn't catastrophic, but it's logged as "rejected" forever and `wiki issues` shows nothing.)

### 2.4 Spec demands ingester wires validator exit code into commit decision; nothing tests this.
`karpathy-wiki-ingest/SKILL.md:398`: *"The ingester MUST NOT call `wiki-commit.sh` until every touched page passes the validator. Wire the validator's exit code into the commit decision; do not paper over it with a log line."* This is prose, not code. The ingester is an LLM following prose. The most likely failure: validator fails, ingester logs the failure to `log.md`, commits anyway because "the page content is mostly correct." Nothing in the test suite exercises a validator-failure-blocks-commit path on a real spawned ingester. (Codex flagged "validator contract drift" for the singular-vs-plural type issue; this is the bigger structural gap.)

### 2.5 `quality.rated_by` enum drift: validator accepts `doctor`, skill says only `ingester` or `human`.
`scripts/wiki-validate-page.py:193`: `VALID_RATED_BY = {"ingester", "doctor", "human"}`. Skill text at `references/page-conventions.md:25`: `rated_by: ingester       # OR "human" — human is sticky, ingester must not overwrite`. There is no `doctor` value in the documented contract. The doctor command isn't implemented (`bin/wiki:44`), so this is a phantom enum value validating future code that doesn't exist. Either document it or remove from validator.

### 2.6 `_index.md` size threshold is enforced via prose, not validator.
`karpathy-wiki-ingest/SKILL.md:338-360` describes a 30-line bash block the LLM is supposed to run after step 7. Each step is conditional and the proposal-pattern existence check is a `find -mtime -1` shell idiom most LLMs will paraphrase wrong. This entire block belongs in `wiki-build-index.py`'s post-write logic (it already loops over indexes; trivial to size-check there). As prose it will rot.

### 2.7 The "always commit a `.gitignore` that ignores `.locks/` and `.ingest.log`" promise is per-wiki only.
`scripts/wiki-init.sh:215-221` writes a wiki-local `.gitignore`. The plugin repo's own `.gitignore` (the modified one in `git status`) is separate. There's no validation that an existing wiki has the gitignore — `wiki-init.sh` checks `[[ ! -f ${wiki}/.gitignore ]]` and skips if present. Pre-v2.4 wikis upgraded to v2.4 likely have a `.gitignore` from the old layout that doesn't ignore `.raw-staging/` (line 61's directory list includes it, but the gitignore content at 215-221 does NOT). `find ~/wiki -name .raw-staging` may show committed staged files in field installs.

### 2.8 `wiki use both` mode persists `fork_to_main = true` but the resolver only reads it from `.wiki-config`. `.wiki-mode` mode never forks.
`scripts/wiki-resolve.sh:117-129` handles `.wiki-mode` and only supports `main-only`. There is no `both-mode` value. If a user manually writes `.wiki-mode = both`, exit 13. This is correct, but the prose at MANUAL.md:38 says *"`wiki use both` — writes `.wiki-config` with fork mode."* — fine — but `wiki use main` is the way to opt out, and there is no `wiki use unfork` or `wiki use no-fork`. Returning to non-fork from fork requires editing `.wiki-config` by hand. No CLI verb covers it.

## 3. The 4-skill split itself

### 3.1 The split is roughly correct, but `karpathy-wiki-ingest` at 428 lines is wrongly sized.
The ingest skill carries:
- 9-step deep orientation (steps 1-9, which is the largest section)
- Capture body-sufficiency check (duplicates structure already in capture-schema.md; should reference, not restate)
- 12-step ingester runbook
- Embedded ~30-line bash for the index-size threshold (item 2.6)
- Tier-1 lint at ingest
- Numeric thresholds
- Three-rule "Category discipline"
- "What's NOT in this skill" sentinel

Steps 1-9 of orientation are operationally distinct from steps 1-12 of the runbook. Two-section split:
  - `karpathy-wiki-ingest/references/orientation.md` — steps 1-9 + cold-start + why-no-embeddings.
  - `karpathy-wiki-ingest/SKILL.md` — body-sufficiency check + 12-step runbook + commit/validator + threshold prose.

The current 428-line file is at the boundary above the 500-line budget v1 set, but is already over the *contextually-loaded-once* budget that motivated the split.

### 3.2 The read skill duplicates the orient procedure that the ingest skill also defines.
`karpathy-wiki-read/SKILL.md` Step A and `karpathy-wiki-ingest/SKILL.md` steps 1-3 both describe "read schema.md, read index.md, read recent log.md entries." They are subtly different (read skill reads `_index.md` per category; ingest skill reads root `index.md` first). The "shape of orient" is duplicated and will drift. Extract to `references/orient-procedure.md` and have both reference it.

### 3.3 The loader claims four iron laws but the read skill has its own subset and the capture skill has none.
`using-karpathy-wiki/SKILL.md:29-34` lists 4 iron laws. The capture skill (`karpathy-wiki-capture/SKILL.md`) has its own "What's NOT in this skill" pointer back to the loader for iron laws, which is correct. The read skill (`karpathy-wiki-read/SKILL.md:9-10`) re-states Iron Rule 4 inline. If the loader iron law list ever changes, the read skill text becomes stale silently.

### 3.4 Capture-side resist-table is missing entirely.
The loader has a *read-side* resist-table (`using-karpathy-wiki/SKILL.md:49-57`). v1's loader had the *capture-side* resist-table that the commit message celebrated as "the REFACTOR-anticipating work" (`tmp/skill-v1-6ce0c77.md:194-209`). The capture skill itself has no such table; the loader has only read-side. **Re-opened rationalizations** the v1 capture-side table forbid:
  - "The user will remember this" — not warned against anywhere in the v2.4 split.
  - "I'll capture it later" — gone.
  - "I'm in the middle of another task; capture is intrusive" — gone.
  - "It's already covered" (without orienting first) — partially covered by Iron Rule 4 for QUESTIONS, but not for capture-trigger moments.
  - "The user didn't ask me to save this" — gone.
  - "I don't have a memory tool available" — gone.
  - "The file is already in a good place (filing ≠ capturing)" — gone.

These need to land somewhere — either in the loader as a second resist-table, or in `karpathy-wiki-capture/SKILL.md` parallel to the read skill's pattern.

### 3.5 No skill claims "queries/" or "ideas/" pages.
`scripts/wiki-init.sh:61` creates `concepts entities queries ideas` directories. `schema.md:134-137` describes them. v1 mentioned: *"If the answer synthesizes across multiple pages in a new way → capture it as a query page to `queries/`."* (`tmp/skill-v1-6ce0c77.md:167`). v2.4 has no skill that tells the agent when to write a `queries/` page or an `ideas/` page. Read skill mentions `queries/` only in the header type list (page-conventions.md:9) and not in any procedural step. Result: the directories exist but the agents have no trigger to populate them. They become dead categories.

### 3.6 The ingest skill's "auto-init decision tree" (v1's branches 1-3) is gone from skill prose.
v1 SKILL.md lines 40-57 had explicit decision logic the agent could follow. v2.4 moved this into `bin/wiki capture`'s exit-10 handler and `wiki-resolve.sh` exit codes. The agent reading the loader does not see when/why each branch fires. The result is the field-test bug where the user typed "wiki init" and the agent had no skill prose mapping that intent to `wiki init-main`. TODO 0.2.9 already tracks the in-active-repo prompt issue but not the broader "where did the decision tree go" concern.

## 6. Hook edge cases

### 6.1 `set -e` is intentionally absent from `hooks/session-start` (line 19). Combined with `|| true` and `|| log_warn ... (non-fatal)` patterns at lines 354, 377, 396, every step silently absorbs failures. There is no escalation if the hook degrades. After 100 sessions where `_drift_scan` always fails, the user gets no signal — only `tail -f .ingest.log` reveals it.

### 6.2 Subagent guard is `CLAUDE_AGENT_PARENT` only (line 31). If Claude Code rebrands the env var (it has changed names in past major versions per a comment in `hooks/stop:7-9`), the fork-bomb guard fails open: every subagent runs the full hook, drains pending captures from the project, etc. The hook should also check `CLAUDE_SUBAGENT`, `CLAUDE_TASK`, or any existing-list variant — at minimum a regex over `env`. The skill `using-karpathy-wiki/SKILL.md:7-9` has its own `<SUBAGENT-STOP>` tag which only the LLM honors; the deterministic env check is the load-bearing guard.

### 6.3 `wiki_root_from_cwd` walks up via parent dir, but ALSO checks `${dir}/wiki/.wiki-config` at every level (`scripts/wiki-lib.sh:57-60`). This means in `/Users/me/projects/A/subdir/`, if `/Users/me/wiki/.wiki-config` exists (the main wiki), the walk-up at depth=2 (`/Users/me/`) finds `/Users/me/wiki/.wiki-config` and returns it as the project's wiki root. That's almost certainly wrong — the user's project is `projects/A`, not `~/wiki/`. The walk should NOT subdir-probe at every level; only at the leaf. Subtle, surfaces as "captures from a project go to the wrong wiki when the project lives under $HOME and has no `.wiki-config` of its own." Codex didn't flag this; MANUAL.md:108-110 doesn't describe this case.

### 6.4 `_reclaim_stale` (hook line 383-396) reclaims `.processing` files >10 min old by renaming back to `.md`. But the original `.md` may have been written by `wiki capture` and the ingester might be in step 5 (decide) when reclaim fires. The reclaim creates a new `.md` while the ingester still holds the `.processing` and is mid-write. Now there are TWO files. Ingester finishes, archives `.processing`, but the reclaimed `.md` lingers and gets re-spawned. The 10-minute threshold is a guess; an ingester orienting against a 50-page wiki easily takes 4-8 minutes. Slow LLM = duplicate ingest.

### 6.5 The hook's loader injection at line 78-85 emits `additional_context` for Cursor based on `CURSOR_PLUGIN_ROOT` being set. But `CURSOR_PLUGIN_ROOT` is the variable a Cursor plugin author sets explicitly. Cursor itself sets `CURSOR_TRACE_ID` and `CURSOR_SESSION_ID`. If this plugin is installed under Cursor without a Cursor-side rename, the `CURSOR_PLUGIN_ROOT` check never fires; it falls through to the Copilot-CLI branch (line 84). The harness signal is wrong. CLAUDE.md says Cursor support is "untested"; this is *why* — the heuristic does not match real Cursor env.

### 6.6 The hook treats absence of `${CLAUDE_PLUGIN_ROOT}` as "not Claude Code", routing to the Copilot/SDK branch (line 79). But `CLAUDE_PLUGIN_ROOT` is set by Claude Code's plugin loader, NOT by `claude -p` invocations directly. The `wiki-spawn-ingester.sh` re-invokes `claude -p` with explicit env passthrough that does NOT include `CLAUDE_PLUGIN_ROOT` (line 57-63). The fork-bomb guard catches this case (WIKI_CAPTURE is set), so it never reaches the JSON branch — but if the guard ever has a bug, every spawned ingester also tries to emit additionalContext as if it were a Copilot CLI session.

## 7. Iron Laws audit

The four iron laws (`using-karpathy-wiki/SKILL.md:29-34`):

1. NO WIKI WRITE IN THE FOREGROUND — clear, verifiable post-hoc by checking that the current process is not the spawned ingester (`WIKI_CAPTURE` set). Agent can self-check.
2. NO PAGE EDIT WITHOUT READING THE PAGE FIRST — ambiguous: how recently? The lock-acquisition contract in `wiki-lock.sh` re-validates by holding the lock; an agent reading the page 3 minutes ago and editing now is "compliant" to the prose. Sharpen: "in the SAME tool-call sequence."
3. NO SKIPPING A CAPTURE BECAUSE "IT DOESN'T LOOK WIKI-SHAPED" — clear in intent but not action-verifiable. There is no log of decisions-not-to-capture; if the agent skipped 4 triggers and captured 1, no one knows. Stop-hook gate (TODO p1 deferred) is the only mechanism that would catch.
4. NO ANSWERING ANY USER QUESTION WITHOUT ORIENTING FIRST — well-actionable (load read skill, run Step A). But the read skill's Step A has the "if you already oriented this session, skip to B" carve-out, which is correct yet creates an "I-already-oriented" rationalization the resist-table tries to forbid. Tension: the skill is BOTH telling the agent to orient AND giving it permission to skip.

Ordering: the laws are listed as a cluster, not prioritized. Reasonable, but Iron Law 1 (foreground writes) is structural and Iron Law 4 (orient-first) is procedural. They should be in two groups, since "must" laws and "must-do" laws fail differently.

## Closing observations

The split was a good call but the split skipped "what's NOT in the loader anymore" — the v1 loader did three jobs (announce, trigger, defend-against-rationalization). v2.4 covers two. The third (capture-side resist-table) was the load-bearing one for "agents who feel the trigger but rationalize past it" — the same failure mode the field-test transcript surfaced when the agent pattern-matched "wiki init" to wiki-status instead of asking what the user meant.

Iron Rule 4 closed an analogous hole on the read side. 0.2.9 should close the same hole on the capture side, plus ship the deterministic-enforcement P0s (validator-blocks-commit, capture-claim-before-spawn, evidence-path-flag).

The single biggest structural risk: **prose is the load-bearing enforcement mechanism for at least 8 contracts** (validator gating, MAX_REJECTIONS, index-size threshold, page-edit-after-read recency, capture-not-skipped, ingester-prepares-orientation, log.md append on every step, single-source-of-truth for mode change). Each one rots the moment the loader skill changes shape and the LLM following the prose reads "differently." The repo's own CLAUDE.md says skills should follow TDD with RED-GREEN-REFACTOR; the runtime tests cover the deterministic shell paths, not the prose contracts.
