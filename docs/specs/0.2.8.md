## Background (read once if needed)

The karpathy-wiki plugin auto-captures durable knowledge into a
git-versioned wiki. Three loops:

- **Capture** — the main agent in any conversation writes a tiny
  `.md` file to `<wiki>/.wiki-pending/` whenever a wiki-worthy moment
  fires (research result, resolved confusion, gotcha, etc.).
- **Ingest** — a detached `claude -p` ingester (spawned by
  `scripts/wiki-spawn-ingester.sh`) reads each capture, orients on
  the wiki, writes/updates pages, validates, commits.
- **Read** — when the user asks a question, the main agent loads
  `karpathy-wiki-read/SKILL.md` and runs a 6-step ladder against the
  wiki before answering.

The plugin runs primarily on Claude Code. A `SessionStart` hook
(`hooks/session-start`) injects the using-karpathy-wiki loader into
every conversation.

The 12 items below are real defects identified in three adversarial
reviews on 2026-05-06. See `docs/reviews/2026-05-06-v0.2.7-synthesis.md`
for the full decision grid (12 ship, 9 deferred, 11 rejected, plus 2
field-test entries on TODO.md and 1 already shipped).

## TDD discipline

Per CLAUDE.md and recent commit pattern: every script change goes
RED-then-GREEN. Two commits per item. Doc-only fixes (items #23 #24
#25) can ship as a single `chore(docs)` commit.

- **RED commit:** new test file under `tests/red/test-<topic>.sh`,
  asserts the target behavior. Test FAILS at this commit. Body of
  commit message names the bug + reproduction.
- **GREEN commit:** the fix. Test passes. Commit body references the
  RED commit SHA.
- **No squash.** Per-step rollback granularity (per CLAUDE.md memory
  feedback).

### Tier 2 — small RED-GREEN fixes (no dependencies)

#### #5: spawn prompt points at deleted monolith

- **Smoking gun:** `scripts/wiki-spawn-ingester.sh:51-53`
- **Bug:** Spawn prompt instructs the ingester to load
  `skills/karpathy-wiki/SKILL.md` — a file deleted in v2.4. Every
  spawned ingester gets a stale skill reference.
- **RED test (`tests/red/test-spawn-prompt-references-current-skill.sh`):**
  Generate a spawn prompt (via the spawner's prompt-construction code
  path, or by reading the spawner script and extracting the prompt
  template). Assert the prompt does NOT contain
  `skills/karpathy-wiki/SKILL.md` and DOES contain
  `skills/karpathy-wiki-ingest/SKILL.md`.
- **GREEN fix:** Update lines 51-53 of `wiki-spawn-ingester.sh` to
  point at the split skill. Verify spawned ingesters load the right
  skill.
- **Verify:** `bash tests/integration/test-end-to-end-plumbing.sh` — should still pass.

#### #11: subdir capture parent-walk regression

- **Smoking gun:** `scripts/wiki-resolve.sh:15-17, 61-65`; doc lie at `MANUAL.md:108-110`
- **Bug:** Resolver only checks `$(pwd)`, doesn't walk up. MANUAL.md
  promises subdir captures route to the project wiki. v1 baseline at
  `tmp/skill-v1-6ce0c77.md:44-46` had the parent-walk; lost in v2.4.
- **RED test (`tests/red/test-resolver-walks-up.sh`):** Create a temp
  project layout: `<tmp>/proj/.wiki-config` (project wiki) plus
  `<tmp>/proj/sub/dir/`. From `sub/dir`, call `wiki-resolve.sh` and
  assert it returns the project wiki path, not the user's main wiki
  or empty.
- **GREEN fix:** Restore parent-walk in the resolver. Walk up until
  finding `.wiki-config` OR hitting `$HOME` OR root. Stop at `$HOME`
  so subdir of `~/proj` doesn't accidentally pick up `~/wiki`.
- **Watch out for:** Item #9 (cross-project leak) is the OPPOSITE
  direction — cwd inside a project must NOT walk up past the project
  into `~/wiki`. Implement #11 with the constraint that satisfies #9.
  Easiest: walk up only inside the user's home tree, stop before
  `$HOME`.

#### #9: cross-project wiki leak

- **Smoking gun:** `scripts/wiki-lib.sh:57-60`
- **Bug:** `wiki_root_from_cwd` walks up AND probes `${dir}/wiki/.wiki-config`
  at every level. From `/Users/me/proj/A/sub`, walk-up reaches
  `/Users/me/` and finds `/Users/me/wiki/`, then routes captures from
  project A to the main wiki.
- **RED test (`tests/red/test-resolver-no-cross-project-leak.sh`):**
  Set up: `~/wiki/.wiki-config` (main wiki, real or temp-faked via
  `$HOME` override) + `<tmp>/proj/A/sub/`. From `sub/`, the resolver
  must return empty or the project's own wiki (if it has one), NOT
  the main wiki.
- **GREEN fix:** Stop probing `${dir}/wiki/.wiki-config` at every
  walk-up level. Probe ONLY at the leaf cwd. Walk-up still works for
  finding `.wiki-config` at parent dirs (the legitimate case from
  #11), but no nested-`wiki/` discovery.
- **Pair with #11.** These are tested together.

#### #12: schema-proposal directory not created at init

- **Smoking gun:** `scripts/wiki-init.sh:60-63` vs ingest skill at
  `skills/karpathy-wiki-ingest/SKILL.md:338-356, 402-421`
- **Bug:** Ingester is told to write to `.wiki-pending/schema-proposals/`
  when a category threshold fires. Init creates `.wiki-pending` but
  not the subdir. First time a threshold fires, the script crashes.
- **RED test (`tests/red/test-init-creates-schema-proposals-dir.sh`):**
  Run `bash scripts/wiki-init.sh main "<tmp>/wiki"`. Assert
  `<tmp>/wiki/.wiki-pending/schema-proposals/` exists.
- **GREEN fix:** Add `mkdir -p "${wiki}/.wiki-pending/schema-proposals"`
  to `wiki-init.sh`.

### Tier 3 — `bin/wiki` evidence-path (schema change, isolated)

#### #1: `--evidence-path` flag; drop hardcoded `"conversation"`

- **Smoking gun:** `bin/wiki:237-258`
- **Bug:** `bin/wiki capture` writes `evidence: "conversation"` for
  ALL kinds, including `chat-attached` and `raw-direct` where the
  schema demands an absolute path. Manifest's iron rule (`origin` =
  capture's `evidence`) is violated for every chat-attached capture.
- **RED test (`tests/red/test-capture-evidence-path.sh`):**
  - Run `echo body | bin/wiki capture --title T --kind chat-only ...`
    — assert capture frontmatter has `evidence: "conversation"` (this
    is correct for chat-only; verifies the test isn't broken).
  - Run `echo body | bin/wiki capture --title T --kind chat-attached --evidence-path /some/abs/path ...`
    — assert capture frontmatter has `evidence: "/some/abs/path"`.
  - Run `bin/wiki capture --title T --kind chat-attached` with no
    `--evidence-path` — assert it fails fast with a non-zero exit
    and a message about chat-attached requiring evidence-path.
- **GREEN fix:**
  1. Add `--evidence-path <abs>` flag to `bin/wiki capture` CLI
     parsing.
  2. Validate: chat-attached and raw-direct REQUIRE
     `--evidence-path` to be set; chat-only IGNORES it (or rejects
     it; pick one).
  3. Write the actual path to capture frontmatter `evidence:` field
     when set, else `"conversation"`.
- **Skill text:** `skills/karpathy-wiki-capture/SKILL.md:17-31` —
  update CLI invocation example to include the new flag.
  `references/capture-schema.md:31-38, 59-64` — re-state the iron
  rule, no code change there.

### Tier 4 — `wiki status` filter (depends on nothing else)

#### #6: `wiki status` filter to actual wiki content set

- **Smoking gun:** `scripts/wiki-status.sh:23, 66-74, 130-145, 153-163`
- **Bug:** `wiki status` counts all `.md` files via glob, including
  `_index.md`, archives, raw artifacts. Page-count metric is silently
  inflated. User reading status sees wrong totals.
- **RED test (`tests/red/test-status-filters-content-set.sh`):**
  - Set up a temp wiki with: 3 real concept pages, 1 root `index.md`,
    2 `_index.md` (one per category), 1 file in `raw/`, 1 file in
    `.wiki-pending/archive/`.
  - Run `bin/wiki status`. Assert "Total pages: 3" (not 7+).
- **GREEN fix:** Define "content set" explicitly. Probably:
  files matching `*.md` UNDER `<wiki>/<category>/` where category is
  in the discovered category list, EXCLUDING `_index.md`. Apply that
  filter in `wiki-status.sh` line 23 and propagate.
- **Watch out:** Quality / drift / tag-synonym counts likely use the
  same broken glob. Fix all metrics consistently.

### Tier 5 — `_raw_recovery` lock window (real race)

#### #8: `_raw_recovery` emit capture inside the lock window

- **Smoking gun:** `hooks/session-start:300-352`
- **Bug:** Function acquires manifest lock, checks manifest, mv's
  file to inbox, releases lock, THEN emits the capture. Window
  between lock release and capture emit allows another SessionStart
  to see the file in inbox and emit a duplicate capture. Or worse:
  the recovery's capture emit references a file path another process
  is racing to claim.
- **RED test (`tests/red/test-raw-recovery-no-duplicate-captures.sh`):**
  Spawn two parallel SessionStarts targeting the same wiki with the
  same unmanifested file in `raw/`. Wait. Assert `.wiki-pending/`
  contains exactly one capture for that file (deterministic file hash
  in the capture filename means it should converge).
- **GREEN fix:** Move the capture-emit (`_emit_raw_direct_capture`)
  inside the lock window. Acquire lock, check manifest, mv file, emit
  capture, release lock.
- **Verify:** `bash tests/integration/test-concurrent-ingest.sh` still passes.

### Tier 6 — LLM-substrate hardening (#3 and #13)

These two were initially rejected as "no reported failure" but
promoted on the principle that the user can't observe LLM-substrate
failures by hand. See synthesis "Ship in 0.2.8" preamble for
reasoning.

#### #3: validator-blocks-commit deterministically

- **Smoking gun:** `skills/karpathy-wiki-ingest/SKILL.md:398`
- **Bug:** Skill says "the ingester MUST NOT call wiki-commit.sh
  until every touched page passes the validator." This is prose
  enforcement only. A sloppy ingester can ignore it; a broken page
  silently lands in the wiki and rots the LLM substrate.
- **RED test (`tests/red/test-commit-blocks-on-validator-fail.sh`):**
  - Create a temp wiki with a malformed page (e.g. missing required
    frontmatter `type` field).
  - Call `bash scripts/wiki-commit.sh "${WIKI_ROOT}" "test commit"`.
  - Assert exit code is non-zero AND no commit was created (`git log`
    unchanged from before the call).
- **GREEN fix:** `wiki-commit.sh` runs `wiki-validate-page.py`
  on every wiki page being committed BEFORE calling `git commit`.
  If validator exits non-zero for any page, commit script exits
  non-zero. The validator already exists and works; we just wire its
  exit code into the commit gate.
- **Skill update:** `karpathy-wiki-ingest/SKILL.md:398` — change
  "MUST NOT call" prose to "the commit script enforces validator
  pass; if validator fails, fix-and-recommit" since enforcement is
  now in code.

#### #13: capture-side resist-table

- **Smoking gun:** `tmp/skill-v1-6ce0c77.md:194-209` (v1 baseline,
  commit 6ce0c77) vs `using-karpathy-wiki/SKILL.md:49-57` (current,
  read-side only)
- **Bug:** v1 had a rationalization-counter table specifically for
  the moment "should I capture this?" The v2.4 split kept a similar
  table but only for the read-side ("orient before answering").
  Capture-side rationalizations like *"the user will remember this"*
  / *"I'll capture it later"* / *"I'm in the middle of another
  task"* / *"the file is already in a good place"* go uncountered.
  Skipped captures are invisible — the user can't observe them.
- **RED test (`tests/red/test-loader-has-capture-resist-table.sh`):**
  Read `skills/using-karpathy-wiki/SKILL.md`. Assert it contains a
  capture-side resist-table by checking for each of the v1
  rationalization rows verbatim or close-paraphrase. Specifically
  check for at least 4 of these 7 v1 rows:
  - "The user will remember this"
  - "It's too trivial for the wiki"
  - "I'll capture it later"
  - "I'm in the middle of another task"
  - "The user didn't ask me to save this"
  - "I don't have a memory tool available"
  - "The file is already in a good place"
- **GREEN fix:** Add a "Resist-table for capture triggers" section to
  `using-karpathy-wiki/SKILL.md` after the TRIGGER section. Two
  columns ("Rationalization" / "Reality"), 7 rows from v1 baseline.
  Place it adjacent to the existing read-side resist-table for
  symmetry.
- **Skill is prose — no functional code change.** RED test asserts
  the prose is present. The mechanism that DEPENDS on this prose
  (agent reading + obeying it) is itself prose; that's the load-
  bearing-prose problem flagged by Opus, not solvable in 0.2.8.
